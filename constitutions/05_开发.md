# 💻 开发宪法

> 开发阶段：基于详细设计，执行 TDD 实现、编码和重构。
> 本阶段通过 TDD 微循环交付 .gd 脚本文件。

## 适用 Skill

| Skill | 职责 | 可修改文件 |
|-------|------|-----------|
| `godot-best-practices` | GDScript 编码最佳实践 | `.gd` 脚本文件 |
| `godot-gdscript-patterns` | GDScript 设计模式优化 | `.gd` 脚本文件 |
| `test-driven-development` | TDD 微循环（Red → Green → Refactor） | `test/**/*.gd` + `.gd` |

## 前置条件

- **P1-18** 开发阶段开始前，**必须**读取以下上游文档，确保实现基于已确认的需求、架构和详细设计：
  - `docs/01_gdd/` 下的需求规格文档（游戏设计文档、功能需求文档）
  - `docs/02_analysis/` 下的需求分析文档（技术可行性分析、性能需求分析）
  - `docs/03_arch/` 下的架构设计文档（架构概要设计、模块设计、状态机设计）
  - `docs/05_guide/` 下的详细设计文档（功能开发指导）
- **P1-18a** 若上述文档不存在，**禁止**进入开发阶段，必须提醒用户先完成上游阶段
- **P1-18b** 代码实现必须与架构设计文档中定义的模块划分、接口规范保持一致

## 规则

### Story 开发流程

- **P1-14** **必须以 task 粒度进行开发** story：
  1. story需要拆分成多个task
  2. 分析story之间的相互依赖关系
  3. 对于没有依赖关系的task可并行进行开发，有依赖关系的task**必须**串行开发
- **P1-15** 每个 Story 开发前，**必须**输出以下内容：
  1. Story 的完整内容（需求描述、验收标准）
  2. AI 对该 Story 的理解和实现思路（包括涉及哪些文件、预期变更范围、技术方案）
  - 输出后暂停，等待用户确认思路后方可开始编码
- **P1-15a** 当 Story 同时涉及场景（`.tscn`）和脚本（`.gd`）开发时，场景框架创建和脚本开发**可并行执行**。AI 创建场景基本框架后，不需要等待用户编辑场景，即可继续编写脚本和测试。当 Story 中所有任务完成后，统一暂停等待用户编辑场景
- **P1-16** 代码开发**必须**使用 TDD 微循环（见下方 TDD 微循环章节），遵循 Red → Green → Refactor 流程
- **P1-17** 代码开发完成后，**必须**进行代码检视（Code Review），检查项包括：
  - 是否符合编码规范（P1-3 ~ P1-5）
  - 是否存在未处理的边界条件
  - 是否有性能隐患
  - 命名是否清晰、逻辑是否易读
  - 是否满足验收标准
- **P1-18** 每个 Story 开发完成后，**必须**检查是否生成了必需的 `.uid` 文件：
  - 新建的 `.gd` 脚本文件必须有对应的 `.uid` 文件
  - 若发现缺失，必须通过 Godot 编辑器重新扫描生成
- **P1-19** 每个 Story 完成后，**必须暂停执行**，向用户汇报完成情况，等待用户确认后再继续下一个 Story

### BDD 行为定义（TDD 前置步骤）

- **P1-1a** 在编写正式测试代码之前，**必须**先遵循 Behaviour-Driven Development 实践，使用 Gherkin schema 编写行为定义：
  1. 使用 `Given-When-Then` 模式为每个测试函数编写行为注释
  2. 创建只包含行为注释和测试函数名（空函数体）的测试文件
  3. 使用 `question` 工具向用户确认行为定义是否正确，获得确认后方可继续
  4. **禁止**在完成行为定义并获得用户确认之前，编写任何测试实现代码

- **P1-1b** 行为定义文件格式示例：
  ```gdscript
  # Given the player has 100 HP
  # When the player takes 30 damage
  # Then the player's HP should be 70
  func test_take_damage_reduces_hp():
      pass

  # Given the player has 10 HP
  # When the player takes 20 damage
  # Then the player's HP should be 0 (not negative)
  func test_take_damage_hp_not_negative():
      pass
  ```

### TDD 微循环

- **P1-1** 使用 `test-driven-development` 技能 + TDD 微循环：
  ```
  BDD 行为定义 (Given-When-Then 空测试文件)
    → question (用户确认行为定义)
    → test-driven-development (编写测试实现)
    → godot-best-practices (实现代码)
    → minimal-godot_get_diagnostics (语法检查)
    → $GODOT_HOME -s addons/gut/gut_cmdln.gd -gexit (运行测试)
    → godot-gdscript-patterns (重构优化)
  ```

- **P1-2** 使用 `context7` 工具查询 API 文档：`context7_resolve-library-id` → `context7_query-docs`

### 编码规范

- **P1-3** 优先 AnimationPlayer 节点
- **P1-4** Singleton .gd 文件禁止 `class_name`
- **P1-5** 禁止三元运算符，使用 `if...else`

### 场景开发约束

#### 总则

- **P1-6** **禁止** AI 创建或修改 `.tscn` 场景文件中需要可视化确认的部分（包括通过 MCP 工具 `godot-mcp_create_scene`、`godot-mcp_add_node`、`godot-mcp_save_scene` 等创建的视觉布局）
- **P1-6a** AI **可以**通过 MCP 工具创建场景的基本框架结构（节点树骨架、基本节点类型），但禁止执行以下操作：
  - 设置精确的位置、锚点、偏移等需要可视化确认的属性
  - 加载图片/纹理等资源（需用户在编辑器中可视化操作）
  - 设置复杂的布局参数（Container 的间距、边距微调等）
  - 配置动画关键帧（需可视化预览确认）

#### AI 与用户的职责分工

- **P1-7** 场景开发采用 **AI 创建框架 + 用户可视化操作** 的协作模式：
  1. **AI 职责**：创建场景基本框架（节点树骨架）+ 输出详细操作指导文档
  2. **用户职责**：所有需要可视化确认的操作（图片加载、位置微调、布局调整、动画配置等）
  3. AI 输出操作指导后，**不阻塞**其他并行任务（脚本编写、测试等），可继续执行 Story 中的其他任务
  4. 场景可视化操作由用户异步完成。当 Story 中所有任务执行完成后，统一暂停等待用户编辑场景（见 P1-7e）

#### 操作指导输出

- **P1-7a** 场景操作指导**必须**以 Markdown 文件形式输出到项目 `.a_manual_guide/` 目录下，文件命名格式：`.a_manual_guide/{两位数序号}_{场景名称}_场景开发指导.md`
- **P1-7b** 场景操作指导文档**必须**包含以下内容（不仅限于属性设置）：
  1. **目标说明**：本次场景操作要达成什么目的，为什么要这么做
  2. **前置准备**：需要提前准备的资源（图片、TileSet、字体等）及其放置路径
  3. **节点树结构**：完整的节点层级树（使用缩进或 ASCII 树表示），每个节点的类型和名称
  4. **详细操作步骤**：逐步指导，每步包含：
     - 操作对象（选中哪个节点）
     - 操作方式（在哪个面板、哪个属性区域操作）
     - 具体值（属性的精确值，如锚点预设名称、坐标、尺寸等）
     - **为什么**这样设置（设计意图，如"左上角锚点确保 HUD 不随窗口缩放偏移"）
  5. **复杂操作专项说明**（适用于以下场景）：
     - **锚点（Anchor）与布局**：解释 Container 的布局规则、anchor preset 的选择理由、margin 与 offset 的区别和设置值
     - **TileSet / TileMap**：TileSet 资源的创建流程、图块分割方式、碰撞形状配置、TileMap 图层绘制顺序
     - **动画（AnimationPlayer）**：轨道说明、关键帧时间点、缓动曲线选择理由
     - **物理碰撞**：碰撞层/掩码的二进制位含义、碰撞形状类型选择理由
     - **信号连接**：信号触发时机、回调函数签名、连接模式（OneShot / Deferred 等）选择理由
  6. **验证方法**：操作完成后如何在编辑器中验证正确性（运行场景、检查 Inspector、预览动画等）
  7. **常见问题**：该场景操作中容易出错的地方及解决方法

#### 串行执行与子agent限制

- **P1-7c** 场景创建/编辑**不需要**暂停其他并行任务。AI 创建场景基本框架后，可继续执行 Story 中的脚本编写、测试等任务
- **P1-7d** 场景开发任务**禁止**在子agent（`task()` 委托）中执行。场景操作指导必须由主会话（Sisyphus）直接输出，确保与用户的交互可控且可追溯
- **P1-7e** 当 Story 中的所有任务都执行完成后，AI **必须暂停执行所有任务**，等待用户在 Godot 编辑器中编辑场景（包括后续 Story 开发、代码编写、测试执行、重构等），直至用户**显式确认**场景编辑已完成
- **P1-7f** 用户显式确认场景编辑完成后，AI **必须**执行以下验证步骤：
  1. 检视场景文件（检查节点树结构、信号连接、脚本挂载是否正确）
  2. 运行相关测试用例验证功能正确性
  3. 确认无误后方可继续下一个 Story

### 代码重构

- **P1-11** 重构前分析：
  ```
  godot-ultimate_godot_document_symbols (理解文件结构)
    → godot-ultimate_godot_find_references (查找所有引用)
    → godot-ultimate_godot_check_patterns (检查代码模式)
    → godot-ultimate_godot_auto_fix (自动修复，默认 dry-run)
  ```

### 性能优化

- **P1-13** 性能优化流程：
  ```
  minimal-godot_get_diagnostics (排除语法问题)
    → minimal-godot_scan_workspace_diagnostics (全项目扫描)
    → godot-ultimate_godot_check_patterns (定位性能模式，如 _process 中的对象创建)
    → godot-mcp_run_project + godot-mcp_get_debug_output (运行时分析)
  ```
